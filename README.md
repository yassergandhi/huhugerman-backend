# huhugerman-backend

**Identity resolution engine for the huhuGERMAN platform.**

A production backend serving 30–40 students per trimester at UAM Azcapotzalco — operational since 2011 in its original GAS implementation, migrated to Supabase when AI feedback integration required POST endpoints.

Built to solve a problem that authentication systems don't solve: *the same student arriving through different identifiers across platforms.*

---

## The Problem Authentication Doesn't Solve

Mexican university students submit assignments through multiple channels. A student registered in the institutional system as `ana.garcia@alumno.uam.mx` may submit from `anagarcia@gmail.com` via a Play Store account, then from `ana.g@hotmail.com` on a shared computer.

**Same person. Three identities in the spreadsheet. One student who can't access their feedback.**

Login systems don't help here. Adding authentication to a formative language exercise platform would introduce:

- **Friction at submission time** — the pedagogy depends on low-friction entry
- **A registration step** that breaks the classroom flow
- **Zero actual identity resolution** — students forget passwords; institutions reset credentials

The real problem is **correlation without custody.** The system needs to recognize that three different email addresses belong to the same student, without requiring the student to authenticate, without requiring the instructor to manually merge records.

---

## Architecture Decisions: Four ADRs

### ADR-001: No Login System

> "Adding authentication solves access control, not identity resolution. We don't need access control. We need correlation."

**Decision:** Collect email as a conscious identifier field. Remove Google's auto-populated account email from form defaults.

**Rationale:** Students are not trying to hide their identity — they are trying to submit quickly. The friction of authentication prevents submission, which breaks the pedagogical flow. The system's job is to recognize that `ana.garcia@alumno.uam.mx`, `anagarcia@gmail.com`, and `ana.g@hotmail.com` are the same person.

### ADR-002: Backend-Generated UUID

> "The student's UUID must be immutable and never exposed to the student or the instructor."

**Decision:** UUID generated by the backend at first valid submission. Not derived from the email. Persists regardless of email changes.

**Rationale:** The UUID is the system's internal identifier. If a student changes their email address mid-semester, their UUID remains constant. Their feedback history remains accessible. The instructor sees the student's name (editable, human-readable); the system sees the UUID (immutable, permanent).

### ADR-003: SHA-256 Fingerprint for Correlation

> "The fingerprint is not a security mechanism. It is a correlation tool."

**Decision:** Normalized email → SHA-256 → stored alongside UUID. When a new submission arrives with a known fingerprint but different surface email (capitalization, alias), the system matches to the existing student record rather than creating a duplicate.

**Rationale:** Email addresses vary in capitalization and domain aliases. `García` vs `Garcia`. `gmail.com` vs `gamil.com` (typo). The fingerprint is deterministic — the same normalized email always produces the same hash.

### ADR-004: Flags Instead of Blocks

> "A system that depends on human judgment not to break is not a product."

**Decision:** Ambiguous submissions receive a `⚠️ REVISAR NOMBRE` flag visible to the instructor. The system never silently rejects a submission. The instructor resolves ambiguity; the system documents it.

**Rationale:** Edge cases exist. A typo might make a fingerprint unrecognizable. The system's job is not to make the final decision — it is to flag ambiguity and let the instructor decide. Every flagged submission is logged for audit.

## Identity Resolution Layers
```
Human layer        → Display name (editable, instructor-visible)
System layer       → UUID (immutable, never exposed)
Correlation layer  → SHA-256 fingerprint (matching across alias/capitalization)
```

Each layer serves a different purpose:

- **Human layer:** The instructor sees `Ana García` and recognizes the student
- **System layer:** The backend tracks submissions, feedback, and history by UUID
- **Correlation layer:** The fingerprint matches new submissions to existing records

## Normalization Logic

Before identity resolution, all inputs pass through normalization:
```typescript
interface NormalizationStep {
  input: string;
  step: string;
  output: string;
  level: 'INFO' | 'WARN' | 'AUTO-FIX';
}

// Example normalization pipeline:
// [INFO]     Input: "ana.garcia@GMAIL.COM"
// [AUTO-FIX] Lowercase: "ana.garcia@gmail.com"
// [INFO]     Fingerprint: "sha256(ana.garcia@gmail.com)"
// [WARN]     Matrícula missing — flagged for instructor review
```

Normalization rules:

- **Spanish capitalization:** Preserve accents (`García` not `Garcia`)
- **Domain correction heuristics:** `gamil.com → gmail.com`, `hotnail.com → hotmail.com`
- **Matrícula validation:** Numeric strings only, or `EXTERNO` for non-institutional students
- **Semantic logging:** Each correction is logged with level (`INFO`, `WARN`, `AUTO-FIX`)

## Migration: Google Apps Script → Supabase

The system began as a pure Google Apps Script solution against Google Sheets. It worked reliably for 6+ years. The migration to Supabase PostgreSQL was triggered by a single constraint: **the AI feedback integration required POST endpoints**, which GAS cannot serve reliably at production latency.

| Capability | Google Apps Script | Supabase PostgreSQL |
|---|---|---|
| Identity resolution | ✅ Full implementation | ✅ Migrated |
| Normalization pipeline | ✅ In-script | ✅ Edge functions |
| POST endpoints | ❌ Unreliable latency | ✅ Native, <200ms |
| AI feedback persistence | ❌ Blocked (no JSONB) | ✅ JSONB columns |
| Cost at current scale | Free | Free tier sufficient |
| Audit trail | ✅ Google Sheets | ✅ PostgreSQL logs |

**Why the migration succeeded:** The identity resolution logic was already proven across multiple trimesters. The migration was not about redesigning the system — it was about moving proven logic to infrastructure that could support POST endpoints.

## Why This Demonstrates Learning Systems Architecture

A developer builds a database. A learning systems architect builds a system that recognizes that the same student is the same student, even when they arrive through different doors.

**Pedagogical constraints shape architecture:** The system doesn't require authentication because the pedagogy doesn't support it. Low-friction submission is a feature, not a bug. The architecture adapts to the pedagogy, not the reverse.

**Correlation without custody:** The system recognizes identity without controlling access. This is the opposite of authentication systems, which control access but don't solve correlation.

**Flags as product:** The `⚠️ REVISAR NOMBRE` flag is not an error state — it is a design decision. The system documents ambiguity and lets humans decide. This is more robust than a system that silently creates duplicates.

**Normalization as pedagogy:** Semantic logging of corrections (`[INFO]`, `[WARN]`, `[AUTO-FIX]`) is not just for debugging. It documents the system's reasoning. An instructor reviewing the logs can understand why a submission was matched to a particular student record.

**Proven logic, proven infrastructure:** The identity resolution logic worked for 6+ years on Google Apps Script. The migration to Supabase was not a rewrite — it was moving proven logic to infrastructure that could support new features. This is the difference between technical debt and technical evolution.
